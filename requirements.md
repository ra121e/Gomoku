# 42-Transcendence-Gomoku 要件定義

## 1. 目的

本ドキュメントは、`42-Transcendence-Gomoku` を「ブラウザで遊べるオンライン五目並べ / 連珠ゲーム」として実装するための要件を定義する。

要件整理のベースは次の 2 点である。

- 現在のリポジトリ構成と既存実装
- 参考サービス `gomoku.com/ja` の公開情報と画面構成

## 2. 現在のリポジトリ把握

### 2.1 現在の技術構成

- アプリ本体: リポジトリルートの Next.js アプリ
  - Next.js 16
  - React 19
  - Bun で起動するカスタム Next.js サーバー
  - Socket.IO クライアント / サーバー
  - Prisma
- データベース:
  - PostgreSQL 16
- ローカル実行:
  - `docker-compose.yml` による `app / database` の 2 サービス構成

### 2.2 現在実装されていること

- 単一の Next.js アプリから `/api/health` を確認できる
- アプリから PostgreSQL への接続確認ができる
- Socket.IO の接続確認ができる
- アプリから `welcome` と `heartbeat` を送信できる
- `/signup` と `/login` が Next.js の form actions で動作し、`/account` でセッション確認できる
- `/api/auth/*` と `/api/matches` の Route Handler が存在する
- Prisma は認証、マッチ、プロフィール、会話の基礎スキーマと migration を持つ

### 2.3 現在未実装の主要領域

- 実際の五目並べ盤面
- 対局ルール判定
- マッチメイキング
- ルーム管理
- ユーザープロフィール
- 対局履歴
- ランキング
- AI 対戦
- チャット

### 2.4 現状から読み取れる前提

- リアルタイム対戦は Socket.IO を中心に設計するのが自然
- 永続データは Prisma + PostgreSQL で管理する前提
- UI、Route Handler、Server Action、Socket.IO サーバーを単一の Next.js アプリに集約する方針
- Docker Compose でローカル開発体験を揃える方針がある

- フレンド、メッセージ、実績は拡張フェーズ扱いが妥当

## 4. プロダクトビジョン

本プロダクトは、以下を満たす Web オンラインゲームを目指す。

- ブラウザだけで五目並べを遊べる
- ローカル開発環境を簡単に立ち上げられる
- ソロ対戦とオンライン対戦の両方を提供する
- 五目並べと連珠の 2 ルールを提供する
- プレイヤーの継続利用を促すプロフィール、履歴、ランキングを持つ

## 7. 機能要件

### 7.1 共通ゲーム要件

- 盤面サイズは 15x15 とする
- 石は交点に置く
- 黒が先手、白が後手とする
- 1 ターンに 1 手のみ置ける
- 既に石がある交点には置けない
- 勝敗、引き分け、投了、切断終了を扱える
- 対局状態はサーバー権威で管理する
- クライアントはサーバーが確定した着手だけを反映する

### 7.2 五目並べルール

- 横、縦、斜めのいずれかで 5 連を作った側が勝利
- 6 連以上も勝利として扱う
- 禁じ手はなし

### 7.3 連珠ルール

- 黒のみ禁じ手対象とする
- 禁じ手は少なくとも以下を対象とする
  - 長連
  - 三三
  - 四四
- 黒はちょうど 5 連で勝利
- 白は 5 連以上で勝利
- 黒が禁じ手を打った場合は即敗北
- 学習用途として禁じ手表示アシストをオプション提供する

### 7.4 ソロプレイ

- プレイヤー対 AI で対局できる
- 難易度を複数段階で選べる
- 対局開始前にルール種別を選べる
- ソロ対戦のみ `リスタート` と `一手戻す` を提供する
- AI の応答待ち中は UI 上で状態を明示する

### 7.5 オンライン対戦

- プレイヤーはニックネームまたはログイン済みアカウントで参加できる
- 公開マッチングで対戦相手を自動で探せる
- ルーム ID / URL によるプライベートルーム作成ができる
- 対戦開始前にルール種別を選べる
- 対局開始時に両者へ開始通知を送る
- 各着手はリアルタイム同期される
- 対局終了時に勝敗理由を表示する
- 対戦相手切断時の扱いを定義する
  - 一定時間以内に復帰なら継続
  - 復帰しなければ切断負け

### 7.6 ロビー / ルーム管理

- 現在参加可能な公開ルーム一覧を表示できる
- ルームの状態を管理できる
  - 募集中
  - 対局中
  - 終了
- ルームごとに以下を保持する
  - ルール種別
  - プレイヤー
  - 観戦可否
  - 現在手番
  - 対局開始時刻
- 重複参加や同時着手を防止する

### 7.7 観戦

- 観戦は公開ルームまたは観戦許可済みルームのみ可能とする
- 観戦者は盤面とチャットをリアルタイム閲覧できる
- 観戦者は着手できない
- MVP では読み取り専用観戦でよい

### 7.8 チャット

- オンライン対局ごとにルームチャットを提供する
- 対局開始 / 勝敗 / 切断などのシステムメッセージを流す
- 不正な長文やスパムを避けるため文字数制限を設ける
- 永続化する場合は対局と関連付けて保存する

### 7.9 ユーザー認証 / アカウント

- メールアドレス + パスワードで登録 / ログインできる
- 将来的に OAuth 拡張ができる設計にする
- ログインユーザーはプロフィールを編集できる
- ゲストプレイは許可するが、ランキング反映条件は別途定義する

### 7.10 プロフィール

- 表示名
- 自己紹介
- アバター
- 総対局数
- 勝敗数
- 勝率
- 現在レーティング
- 最近の対局履歴

### 7.11 ランキング / 統計

- 少なくとも以下のランキングを提供する
  - 五目並べ
  - 連珠
- ソート条件
  - レーティング順
  - 勝利数順
- 保存したい統計
  - 総対局数
  - 勝利数
  - 敗北数
  - 引き分け数
  - 勝率
  - 平均対局時間

### 7.12 対局履歴

- 各対局について以下を保存する
  - 参加者
  - ルール種別
  - 開始時刻 / 終了時刻
  - 勝者
  - 終了理由
  - 手順
  - チャットログの有無
- 登録ユーザーは自分の対局履歴を一覧表示できる

## 8. UI / UX 要件

### 8.1 全体方針

- 初見ユーザーが最短で遊べる導線にする
- 1 画面目でモード選択ができる
- 盤面、対局情報、チャット、ルーム状態が見やすく整理されている
- PC とモバイルの両方で操作できる

### 8.2 トップページ

- ゲーム概要を表示する
- 次の導線を明確に配置する

  - オンライン対戦
  - ランキング
  - ログイン / 登録
- システム疎通確認ページから最終的にゲームホームへ置き換える

### 8.3 対局画面

- 盤面中央配置
- 両プレイヤー情報表示
- 手番表示
- ルール種別表示
- チャット表示
- 対局結果モーダルまたはパネル表示
- 再戦またはロビー復帰導線

### 8.4 アクセシビリティ

- 色だけに依存せず手番や状態を伝える

- 文字サイズとコントラストを確保する

## 9. 技術要件

### 9.1 アーキテクチャ

- リポジトリルート
  - 画面表示
  - 認証 UI
  - ロビー UI
  - 対局 UI
  - Route Handler / Server Action
  - Socket.IO クライアント / サーバー
  - 対局状態管理
  - Prisma を通じた永続化

### 9.2 リアルタイム通信

- マッチメイキング、着手、チャット、切断通知は Socket.IO を使う
- サーバー側でイベント名と payload スキーマを統一する
- 少なくとも以下のイベントが必要
  - `queue:join`
  - `queue:matched`
  - `room:join`
  - `game:start`
  - `game:move`
  - `game:update`
  - `game:end`
  - `chat:send`
  - `chat:receive`
  - `presence:update`

### 9.3 データモデル要件

最低限、以下のモデルが必要である。

- `User`
- `Session` または認証トークン管理用モデル
- `Profile`
- `GameRoom`
- `Game`
- `GamePlayer`
- `Move`
- `Rating`
- `LeaderboardSnapshot` または集計用 view
- `ChatMessage`
- `Presence`

### 9.4 サーバー権威ロジック

- 着手合法性は必ずサーバーで判定する
- 勝敗判定は必ずサーバーで行う
- クライアント側判定は補助表示に留める
- 再接続時はサーバーの最新状態を再送する

### 9.5 認証 / セキュリティ

- パスワードはハッシュ化して保存する
- 入力値検証を API と Socket.IO の双方で行う
- レート制限を設ける
- チャットは XSS を防ぐ
- CORS と Cookie / Token 方針を明確にする

## 10. 非機能要件

### 10.1 性能

- 通常時の着手反映は体感で即時に見えること
- 同時接続数が少人数でも不安定にならないこと
- 対局中にフロントが過度に再描画しないこと

### 10.2 可用性 / 耐障害性

- DB 未接続時に適切なエラーが返ること
- ソケット切断時に再接続を試みること
- 対局中断時の扱いを一貫させること

### 10.3 保守性

- ルール判定ロジックは UI から分離する
- 五目並べと連珠の差分を拡張しやすい構造にする
- テストしやすいドメイン層を持つ

### 10.4 テスト

- 少なくとも以下をテスト対象にする
  - 勝敗判定
  - 禁じ手判定
  - マッチメイキング
  - 切断 / 再接続
  - 認証
  - ランキング集計

## 14. MVP 完了条件

以下を満たしたら MVP 完了とみなす。

- 2 人のプレイヤーがブラウザから同じ対局に入り、リアルタイムで五目並べを最後まで遊べる
- 勝敗が正しく判定される
